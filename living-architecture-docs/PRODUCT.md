# Product

## Register

brand

## Users
Clinicians and research collaborators reviewing the Retina CAD project. They need a fast, credible explanation of what the repo already does, what is still roadmap, and how the data and model pipeline fit together.

## Product Purpose
This surface explains the retina OCTA CAD project in plain language. It should help collaborators understand the current implementation status, trust the rigor of the data pipeline, and distinguish shipped work from planned work without reading the notebook or the full plan.

## Brand Personality
Calm, clinical, futuristic.

Voice should feel precise, confident, and technically serious, but still readable for non-ML specialists. The emotional goal is trust first, intrigue second. Reference direction: Moonshot AI, especially the dark atmosphere, sparse composition, premium restraint, precise typography, and subtle futuristic energy.

## Anti-references
Do not make this feel like a salesy SaaS landing page, a busy dashboard, or a playful consumer AI site. Avoid flashy gradients, infographic clutter, over-carded layouts, loud promotional copy, and anything that blurs the line between implemented results and future ambitions.

## Antipatterns
- Repeated card grids as the default page grammar. Cards are generic, flatten hierarchy, and make the explainer feel like a dashboard.
- Metric-card walls, feature-card rows, or any layout where each idea is trapped in the same boxed module.
- A large rounded bordered shell that wraps an entire section still counts as a card. Simulator shells, compare shells, glossary shells, and similar section-wide wrappers are part of the same failure mode.
- Diagram nodes rendered as uniform rounded UI cards by default. In this project, flow diagrams should read like structure and sequence first, not like a dashboard of widgets.
- If a section can be summarized as “three cards,” “two compare cards,” or “a big card containing smaller cards,” stop and rewrite it with rhythm, rules, columns, and spacing instead.
- Nested containers whose only job is decoration rather than structure.
- Diffuse blue atmospheric background washes behind the hero or major sections. They read like generic sci-fi AI wallpaper, compete with the typography, and shift the tone from clinical briefing to tech mood-board.
- Pill-shaped chips used as default UI grammar for navigation, labels, filters, or status notes. They make the explainer feel productized and dashboard-like instead of calm, editorial, and structural.
- Any visual treatment that makes roadmap items feel as concrete or proven as implemented evidence.

## Design Principles
1. Explain rigor, not hype: show careful method and honest status.
2. Separate present from future: implemented work and roadmap must never be visually or verbally conflated.
3. Make technical depth legible: translate ML concepts into collaborator-friendly mental models.
4. Signal frontier research with restraint: advanced, futuristic tone, but calm and controlled.
5. Preserve scientific trust: every design choice should support credibility, clarity, and reviewability.
6. Use containers sparingly: structure should come from rhythm, alignment, and spacing before boxes.

## Accessibility & Inclusion
Target WCAG AA. Support reduced motion. Maintain strong contrast on dark surfaces, especially body text and secondary text. Use plain-language labels where possible so non-ML collaborators can follow the narrative.
